# v1.1 Security Hardening Design

**Date:** 2026-03-28
**Scope:** gRPC mTLS with hub-issued short-lived certificates + httpOnly cookie auth with CSRF protection

---

## 1. gRPC mTLS with Hub-Issued Certificates

### 1.1 Hub CA

On first boot, the hub generates an ECDSA P-256 CA keypair:

- Self-signed CA cert with 10-year validity
- CA cert and encrypted private key stored in a new `ca_config` SQLite table
- Private key encrypted using a key derived from the existing JWT secret
- CA cert also written to `/data/ca.crt` for manual distribution if needed

### 1.2 Agent Registration (Certificate Bootstrap)

The initial registration flow extends the existing token-based handshake:

1. Agent connects using the existing token handshake (over Traefik TLS or direct plaintext for LAN)
2. Agent generates an ECDSA P-256 keypair locally and creates a CSR
3. Agent sends `RegisterAgent` with the CSR in a new `csr` field (DER-encoded)
4. Hub validates the token, signs the CSR with the CA key, producing a 12-hour client cert with `CN=<serverID>`
5. Hub returns `RegisterAck` with the signed cert and CA cert
6. Agent stores keypair, cert, and CA cert in `/etc/aerodocs/tls/` (root) or `~/.aerodocs/tls/` (non-root)
7. All subsequent connections use mTLS

The agent's private key never leaves the agent machine.

### 1.3 Certificate Renewal (In-Stream)

Modeled after Teleport's approach — renewal happens over the existing authenticated gRPC stream:

1. Agent tracks cert expiry, triggers renewal at 50% lifetime (6-hour mark)
2. Agent generates a new ECDSA P-256 keypair and CSR
3. Agent sends `CertRenewRequest{csr}` on the existing stream
4. Hub validates the request comes from an authenticated agent (verified by mTLS), signs the CSR
5. Hub returns `CertRenewResponse{client_cert, ca_cert}` (CA cert included to support future CA rotation)
6. Agent atomically swaps the cert files on disk
7. Next reconnection uses the new cert

If renewal fails (hub unavailable), the agent continues operating on the current cert until it expires. The 12-hour window provides sufficient buffer.

### 1.4 Connection Verification

- Hub gRPC server configured with `tls.RequireAndVerifyClientCert` using the CA cert pool
- Hub extracts `CN` from the client cert and validates it matches the `serverID` the agent claims in heartbeats
- `ConnManager.Register` rejects connections where the cert CN does not match the claimed server ID
- Prevents a compromised agent from impersonating another agent's server ID

### 1.5 TLS Auto-Detection Update

The agent's `useTLS()` method is updated:

- If the agent has a stored client cert and CA cert, always use mTLS regardless of hostname/IP
- If no stored certs (first registration), use the existing auto-detection logic (hostname → TLS, IP → plaintext)
- After registration succeeds and certs are received, subsequent connections always use mTLS

### 1.6 Migration Strategy

- Hub accepts both plaintext and mTLS connections during rollout
- A `--require-mtls` flag (default `false`) controls whether unauthenticated gRPC connections are rejected
- Existing agents without certs continue to work until they are re-registered or upgraded
- Agents that already have a `serverID` but no cert can request a cert via the new `CertRenewRequest` on their next heartbeat reconnect

## 2. httpOnly Cookie Auth with CSRF Protection

### 2.1 Cookie Structure

| Cookie | Flags | Content |
|--------|-------|---------|
| `aerodocs_access` | httpOnly, Secure, SameSite=Strict, Path=/ | JWT access token |
| `aerodocs_refresh` | httpOnly, Secure, SameSite=Strict, Path=/api/auth/refresh | JWT refresh token |
| `aerodocs_csrf` | Secure, SameSite=Strict, Path=/ | Random CSRF token (NOT httpOnly) |

The refresh cookie is path-scoped to `/api/auth/refresh` so it is only sent on refresh requests, reducing exposure.

In development mode (`--dev`), the `Secure` flag is omitted to allow `http://localhost` testing.

### 2.2 Backend Changes

**Login handler:**
- Sets `aerodocs_access`, `aerodocs_refresh`, and `aerodocs_csrf` cookies via `Set-Cookie` headers
- Returns `{user}` in the JSON body (no tokens in body)
- CSRF token is a random 32-byte hex string, validated statelessly by comparing the `X-CSRF-Token` header against the `aerodocs_csrf` cookie value (double-submit cookie pattern — no server-side state needed)

**Refresh handler:**
- Reads refresh token from `aerodocs_refresh` cookie (not request body)
- Sets new access + refresh + CSRF cookies
- Returns 204 No Content on success

**Auth middleware:**
- Reads access token from `aerodocs_access` cookie first
- Falls back to `Authorization: Bearer` header for backward compatibility and non-browser clients
- Context population unchanged (userID, role, tokenType)

**CSRF middleware:**
- Applied to POST, PUT, DELETE, PATCH requests
- Validates that `X-CSRF-Token` header matches the `aerodocs_csrf` cookie value
- GET, HEAD, OPTIONS requests are exempt
- Requests using `Authorization: Bearer` header (non-cookie auth) are exempt from CSRF check

**Logout handler:**
- Clears all three cookies with `Max-Age=0`
- Returns 204 No Content

### 2.3 Frontend Changes

**Token storage removal:**
- Remove `localStorage` usage for tokens (`getAccessToken`, `setTokens`, `clearTokens` in `web/src/lib/auth.ts`)
- Remove `Authorization` header injection from `apiFetch`

**Cookie-based auth:**
- Add `credentials: 'same-origin'` to all fetch calls (browser sends cookies automatically)
- Read CSRF token from `aerodocs_csrf` cookie via `document.cookie` parsing
- Add `X-CSRF-Token` header to all mutating requests (POST/PUT/DELETE/PATCH)

**Auth flow:**
```
Before: Login → JSON {access_token, refresh_token} → localStorage → Bearer header
After:  Login → Set-Cookie headers → browser auto-sends → X-CSRF-Token for mutations
```

**Refresh flow:**
- On 401 response, call `POST /api/auth/refresh` (cookie sent automatically)
- On success (204), retry the original request (new cookie already set)
- On failure, redirect to login

### 2.4 Backward Compatibility

- Auth middleware checks cookie first, falls back to `Authorization` header
- This allows gradual migration and keeps non-browser clients working
- Existing localStorage tokens expire naturally (15 min for access, 7 days for refresh)
- No breaking API changes — endpoints accept both auth methods

## 3. Proto Changes

### 3.1 Modified Messages

```protobuf
message RegisterAgent {
  string token = 1;
  string hostname = 2;
  string ip_address = 3;
  string os = 4;
  string agent_version = 5;
  bytes csr = 6;  // DER-encoded PKCS#10 CSR
}

message RegisterAck {
  bool success = 1;
  string server_id = 2;
  string error = 3;
  bytes client_cert = 4;  // DER-encoded X.509 client certificate
  bytes ca_cert = 5;       // DER-encoded X.509 CA certificate
}
```

### 3.2 New Messages

```protobuf
message CertRenewRequest {
  bytes csr = 1;  // DER-encoded PKCS#10 CSR
}

message CertRenewResponse {
  bytes client_cert = 1;  // DER-encoded X.509 client certificate
  bytes ca_cert = 2;       // DER-encoded X.509 CA certificate
  string error = 3;
}
```

### 3.3 Message Routing Updates

```protobuf
message AgentMessage {
  oneof payload {
    // existing fields 1-8...
    CertRenewRequest cert_renew_request = 9;
  }
}

message HubMessage {
  oneof payload {
    // existing fields 1-11...
    CertRenewResponse cert_renew_response = 12;
  }
}
```

## 4. Database Migration

Migration `008_add_ca_config.sql`:

```sql
CREATE TABLE IF NOT EXISTS ca_config (
  id TEXT PRIMARY KEY DEFAULT 'default',
  ca_cert BLOB NOT NULL,
  ca_key_encrypted BLOB NOT NULL,
  created_at DATETIME DEFAULT (datetime('now'))
);
```

The CA keypair is generated on first access (lazy initialization) and cached in memory for the lifetime of the hub process.

## 5. Testing Strategy

### Unit Tests
- CA generation: valid ECDSA P-256 keypair, self-signed cert, proper extensions
- CSR signing: valid cert produced, correct CN, correct expiry (12h), correct CA chain
- Cert validation: expired certs rejected, wrong CA rejected, CN mismatch rejected
- Cookie setting: correct flags (httpOnly, Secure, SameSite, Path)
- Cookie reading: auth middleware extracts token from cookie
- CSRF validation: matching header accepted, missing/mismatched rejected, GET exempt, Bearer auth exempt

### Integration Tests
- Agent registers → receives cert → disconnects → reconnects with mTLS → heartbeat succeeds
- Agent cert renewal: connect → wait for renewal trigger → receive new cert → reconnect with new cert
- Login → cookies set → API call with cookie → CSRF header validated → success
- Login → mutating request without CSRF header → 403 Forbidden
- Refresh flow: access cookie expired → POST /api/auth/refresh → new cookies → retry succeeds

### E2E Tests
- Full login flow with cookies in Playwright (verify httpOnly cookies are set, CSRF header sent)
- Agent binary connects with mTLS to hub running with `--require-mtls`

---

## 6. Constraints

- No external dependencies (no Vault, no step-ca)
- Preserve existing API contracts — cookie auth is additive, not replacing Bearer auth
- Agent private keys never leave the agent machine
- Hub CA private key encrypted at rest in SQLite
- `--require-mtls` defaults to false for backward compatibility
- CSRF protection does not apply to non-browser clients using Bearer auth
